1. 개요
1.1 문제 정의
사내 MCP 활용 현황

AI, LLM 기반 업무 자동화 도입 확대로 MCP 기반 서버, 툴이 빠르게 증가

각 팀이 자체 개발한 MCP가 사내 레포지토리에 분산

외부 MCP(Server Marketplace), 내부 MCP(Server), 로컬 MCP(Server) 혼재

Cursor(VSCode), Copilot(Amazon Q) 등 개발도구와 MCP 연동 비중 증가

낮은 운영효율 및 확장성

팀 단위의 단발적 MCP 구축: 각 조직이 자체적으로 MCP Server, Tool을 만들고 배포하여 중복 개발 증가

낮은 활용빈도: 모델, 도구, 데이터 자산이 흩어져 있어 검색 및 활용이 어려움

높은 진입 장벽: MCP서버 설치, 연동에 대한 가이드 등이 대부분 개발자 중심으로 적용되어 있고, 분산된 MCP서버 활용의 어려움 존재

1.2 목표
사내 모든 팀이 MCP(Model Context Protocol) 도구와 n8n 노드 또는 템플릿을 활용할 수 있도록 검색, 등록, 실행, 관리할 수 있는 플랫폼 구축

Provider, Customer 모두 활용 가능한 플랫폼

도구를 찾는 곳이 아닌 AI로 바로 일을 시작할 수 있는 플랫폼

AS-IS: 팀별로 서버, 툴, 데이터를 각각 관리하고 각 팀내에서 MCP 별도 구성하여 안내 및 활용

각 팀에서 개발 및 활용하여 재사용이 어려운 구조

TO-BE: MCPHUB에서 MCP 서버, 도구, 데이터를 등록하여 공유하여 누구나 쉽게 활용

Provider가 MCP서버, 도구를 등록하고 Customer가 실행이 가능한 구조

MCP 서버를 통해 정보 접근 시 각 사이트의 접근권한 기반으로 결과제공

기본 사용자 역할은 다음 3가지로 구분하지만 모든 사용자는 ‘Provider + Customer’ 권한을 가진다

Provider: MCP 서버, 툴의 등록 및 관리

Customer: MCP 검색 및 활용

Admin: 플랫폼 전체 관리(Provider의 관리 권한 포함)

세부권한은 워크스페이스 단위로 관리한다 
Phase2

등록 및 관리 정책

등록 가능한 유형은 ‘MCP 서버, n8n Template’ 으로 구분한다

등록에 필요한 최소 메타데이터 충족 후 사전 연결 테스트(헬스체크)를 완료해야 게시 가능하다

공개 범위는 아래와 같이 구분하여 접근 권한을 관리한다

공개(Public): 모든 사용자가 확인 및 사용가능

비공개(Deprecated): MCP서버/툴을 제공하지 않아 사용 불가능

제한공개(Restricted): 워크스페이스 내에서만 확인 및 사용 가능 
Phase2

임시저장(Draft): 게시되지 않은 상태로 등록자 본인만 확인 가능

실행 및 사용 정책

사용자는 MYMCP에 추가한 MCP만 실행 가능하다

MYMCP사용 상태는 아래와 같이 관리되며 사용상태의 MCP서버만 호출하여 사용가능하다

사용(Active): 사용 가능한 상태

미사용(Inactie): 사용 불가능한 상태

그외 삭제 기능을 통해 MYMCP 목록에서 불필요한 목록을 제고할 수 있다

삭제(Delete): MYMCP 목록에서 제거

보안 정책
